## Audit-Plan: Routing-Regeln vs PDF

### Ziel

Prüfen, ob für jede aktive Routing-Regel (derived variable, REQ-
Regel, CON-Regel) die **Trigger-Bedingung** zur **PDF-Funktion**
des Produkts passt, das sie empfiehlt/excluded.

### Drei Phasen

**Phase 1 — Inventar (Computer-driven, ~30 Min)**

Aus Sheets extrahieren und als Tabelle aufschreiben:
- `map_derived_variables`: pro Variable die regel_json,
  konsumenten, doku
- `map_slot_rules` (aktiv=TRUE): pro Regel trigger_flag(_wert) +
  trigger_flag2(_wert2) + slot_typ + filter + reason
- `map_conflict_rules` (aktiv=TRUE): pro Regel trigger + action +
  match_typ + match_wert
- `map_pflegelevel_scoring`: PFL-Regeln (Pflegelevel-Punktevergabe,
  unkritisch — kein Produkt-Routing)

Output-Format pro Regel: „**Wenn X**, **dann wird Produkt Y für
Slot Z empfohlen**".

**Phase 2 — Plausibilitäts-Check (logisch, ~30 Min)**

Für jede „Trigger → Produkt"-Beziehung ohne PDF-Lesen:
- Macht der Trigger fachlich Sinn?
- Gibt es einen Plausibilitäts-Bruch? (Beispiel: ein Volumen-
  Produkt triggert auf trockenes Haar?)
- Sind irgendwelche Profile-Kombinationen nicht abgedeckt?

Verdachtsfälle markieren (rot/gelb) für Phase 3.

**Phase 3 — PDF-Vertiefung (manuell, nur Verdachtsfälle)**

Pro Verdachtsfall: PDF lesen (Read-Tool), exakte PDF-Aussage
notieren, mit Trigger vergleichen. Wenn Diskrepanz → Fix als
Migration.

### Potenzielle Verdachtsfälle (Hypothesen, vor Audit)

Aus dem Trockenshampoo-Bug-Pattern lassen sich ähnliche Klassen
vermuten:

1. **Inversionen** — Trigger meint das Gegenteil des PDF-Ziels
   (wie der_champ-Fall)
2. **Lücken** — User-Eigenschaft sollte triggern, tut es aber nicht
   (z.B. `gefaerbt` triggert vielleicht nicht needs_repair_focus,
   obwohl Bond-IQ-PDFs auch Färbeschäden adressieren)
3. **Über-Trigger** — Regel feuert in Profilen, für die das Produkt
   PDF-mäßig nicht gedacht ist (z.B. moxie_mousse-Empfehlung bei
   dickem Haar — PDF sagt „feinem bis mittlerem")
4. **Fehlende Negativ-Bedingungen** — Produkt wird empfohlen,
   obwohl ein anderer Eigenschaftswert es ausschließt (z.B.
   smoothing_fohn_spray bei Locken — vor Migration #14 ein Bug)

### Konvention für identifizierte Bugs

Pattern wie bisher: Sheet-Edit (regel_json oder REQ-Trigger),
Backup, Validation per `python3 test_suite.py --profile all
--save`, Diff gegen Pre-Edit-Baseline. Bei Drift: gewollt
dokumentieren.
