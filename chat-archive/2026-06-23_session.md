### 3. JSON-Schema festlegen

User-Entscheidung: JSON statt Mini-DSL. Begründung: kein eigener Parser,
direkt JSON.parse-bar in Node 05, Sheet wird über Agent (nicht manuell)
bearbeitet — Menschenlesbarkeit zweitrangig.

Operator-Inventar (8 Operatoren): `eq`/`neq`, `in`/`nin`, `includes`,
`intersects`, `truthy`, `and`/`or`/`not`, plus `cases` für Enum-Outputs.
Pfad-Syntax als String: `normalized.x` / `flags.x` / `priorities.x`.
Sheet-Reihenfolge = Auswertungsreihenfolge (Phase 1 nur normalized,
Phase 2 mit flags.*). Damit kein Topological-Sort nötig.

3 Schema-Punkte vom User freigegeben:
1. Pfad-Syntax als String (statt verschachteltes Ref-Objekt)
2. Bool-Repräsentation als echtes JSON `true`/`false`
3. R13/R14 (`pflegelevel_final`, `max_products`) komplett rausnehmen
   (Doppel-Doku zu `map_pflegelevel_*`-Tabs)

## Wichtige Erkenntnisse

1. **Sheet-Reihenfolge als implizite Topologie ist praktikabel.** Statt
   einer Engine, die regel-Abhängigkeiten via Topological-Sort auflöst,
   reicht eine sequentielle Auswertung in Sheet-Reihenfolge. Phase 1 vor
   Phase 2, innerhalb Phase 2 keine Forward-Refs erlaubt. Vorteil: Edit-
   Drift (z.B. wenn jemand eine Zeile verschiebt) wird zum sauberen
   Engine-Error („Flag nicht gefunden") statt zu stillem Fehlverhalten.
   Voraussetzung: Konsumenten-Spalte korrekt halten, damit Reihenfolge
   nicht versehentlich kippt.

2. **`volume_sensitivity` war Code-Leiche, drei weitere Flags sind
   Cleanup-Kandidaten.** Sheet- und Code-Grep zeigten 4 Flags ohne
   Konsumenten: `volume_sensitivity` (byte-identisches Duplikat zu
   `needs_lightweight_logic`), `needs_detangling`, `styling_goal_natuerlich`,
   `needs_protection_focus`. Eliminiert wurde nur das Duplikat — die
   anderen drei sind semantisch eigenständige Heuristiken, die nur derzeit
   keine REQ-/CON-Regel triggern. Bei semantik-erhaltender Migration
   bleiben sie drin, im `konsumenten`-Feld markiert. **Separate Cleanup-
   Welle nach Bestätigung der Konsumenten-Lücke.**

3. **n8n REST API ist OpenAPI-streng.** Settings-Felder, die n8n-Cloud im
   GET zurückgibt (`availableInMCP`, `binaryMode`, `timeSavedMode`), werden
   im PUT abgelehnt. Selbe Asymmetrie auf Top-Level (`activeVersion`,
   `nodeGroups`, `sourceWorkflowId`, `staticData`). **Whitelisting im
   Patch-Skript war Pflicht, nicht Best-Practice** — sonst 400er. Konvention
   für künftige Workflow-Patches: PUT-Body auf `{name, nodes, connections,
   settings}` reduzieren, settings auf das API-Schema filtern.

4. **T-03-Workaround ist jetzt etabliertes Standard-Pattern.** Test-Suite-
   Polling timed bei jedem Slot-Drift-Lauf out (90s vs. 3-5 Min reale
   Latenz), aber Direct-API-Read der success-Executions liefert in <30s
   exakte `final_routine`-Daten zur Diff-Bildung. Vier Sessions in Folge
   (2026-06-16, 2026-06-18, 2026-06-22, heute) liefen so. **Folge:
   `test_suite.py --save` produziert seit Wochen keine brauchbaren
   Baseline-Files mehr — bisherige test_results_*.json haben alle
   `output=None`.** T-03-Formalisierung (Polling-Limit hoch oder Direct-
   API-Default) bleibt offener Folgepunkt.

5. **Engine-Selbstkontrolle in Python vor JS-Deploy lohnt sich.** Der
   Python-Port der JS-Engine ist 1:1 mappbar (gleiche Operator-Semantik,
   gleiche Schema-Parsing) und braucht keine n8n-Globals. Stress-Test-
   Profil mit Multi-Bedingungen + Phase-2-Cascade lieferte Confidence,
   dass die Logik-Übertragung von Inline auf Sheet+Engine korrekt war —
   bevor der Production-Workflow angefasst wurde. Pattern für künftige
   Sheet-getriebene Migrationen.
