### 4. Migration #2: Node 12 Filter-Logik generisch

**Audit-Vorlauf (Phase 2a):** Aus Execution 177 die echten `map_slot_rules`-Daten
gezogen. Befund: nur **3 von 25 aktiven Regeln** nutzen die hardcoded Filter-Strings,
und einer der vier (`ir_clinical_serum`) war **toter Code** (0 Treffer).

| Regel | Filter-Wert | Effekt |
|---|---|---|
| REQ-02 (Sarah-Spülung) | `ist_bonding=TRUE` | echte Boolean-Logik |
| REQ-03 (Sarah-Maske) | `ist_bonding=TRUE\|reparatur` | echte Boolean-Logik |
| REQ-04 (Lena/Vivien/Sarah styling_1) | `ist_hitzeschutz=TRUE` | echte Boolean-Logik |
| REQ-05 (Bianca-Kopfhaut) | `scalp_comfort` | bereits durch generischen Substring-Match abgedeckt |
| – | `ir_clinical_serum` | **0 Nutzung — toter Code** |

**Entscheidung:** Keine Sheet-Änderungen. Nur Node-12-jsCode patchen.

**Code-Änderung in Node 12:**

```diff
-    filters.some(f =>
-      prod.produkt_key?.includes(f) ||
-      prod.produktlinie?.includes(f) ||
-      prod.hauptfunktion?.includes(f) ||
-      (f === 'ist_bonding=TRUE' && boolVal(prod.ist_bonding)) ||
-      (f === 'ist_hitzeschutz=TRUE' && boolVal(prod.ist_hitzeschutz)) ||
-      (f === 'scalp_comfort' && prod.produktlinie === 'scalp_comfort') ||
-      (f === 'ir_clinical_serum' && prod.produkt_key === 'ir_clinical_kopfhautserum')
-    )
+    filters.some(f => {
+      const flagMatch = f.match(/^([a-zA-Z_][a-zA-Z0-9_]*)=(TRUE|FALSE)$/);
+      if (flagMatch) {
+        const field = flagMatch[1];
+        const expected = flagMatch[2] === 'TRUE';
+        return boolVal(prod[field]) === expected;
+      }
+      return prod.produkt_key?.includes(f)
+          || prod.produktlinie?.includes(f)
+          || prod.hauptfunktion?.includes(f);
+    })
```

- 2 tote Special-Cases entfernt
- 2 echte Logik-Cases durch generischen `<feld>=TRUE|FALSE`-Regex-Parser ersetzt
- jsCode: 3796 → 3988 Zeichen (+192)

**Regression:** Exec 178–184, **7/7 identisch** zur Baseline.

**Skalierbarkeits-Bonus:** Jede Boolean-Produkteigenschaft kann jetzt ohne Code-Anfassen

- Node 06 jsCode: 4361 → 6409 Zeichen (Phase 4 + Phase 5 raus, generischer Auswerter rein)

**Implementierungs-Detail — Else-If-Semantik erhalten:**

Die alte Phase 4 war eine `if / else if / else if`-Kette: nur die erste passende
Regel feuert, danach wird die Kette abgebrochen — egal ob `levelNum < target` schon
erfüllt war. Im neuen Code wird das nachgebildet durch:

- Raise-Regeln nach `prioritaet` sortiert
- Erste Regel mit passenden Bedingungen → `raiseMatched` setzen + `break`
- Update nur, wenn `levelNum < ziel_levelNum`
- Cap-Regeln greifen **nur wenn `raiseMatched === null`** (keine raise-Regel matched —
  auch nicht ohne Update)

Damit bleibt das Sarah-Edge-Case-Verhalten (blondiert + stark_gesch + heat,
levelNum bereits HIGH) korrekt: rule 1 matched (bricht Kette), aber kein Update;
cap wird übersprungen, weil raiseMatched gesetzt ist.

**Regression:** Exec 185–191, **7/7 identisch** zur Baseline.

**Performance-Beobachtung:** Polling pro Profil jetzt 12–16 statt 6–10 (ein zusätzlicher
Sheet-Loader-Round-Trip), aber alle innerhalb der 15s-Grenze.

### 7. Test-Suite-Härtung (drei strukturelle Änderungen)

Anlass: der Latenz-Fehler oben. Änderungen in `test_suite.py`:

```diff
- DEFAULT_MAX_WAIT = 15
+ DEFAULT_MAX_WAIT = 90

- TERMINAL_STATUSES = ("success", "error", "crashed", "failed", "canceled")
+ TERMINAL_STATUSES = ("success",)
```

Und neu eingeführt: `fetch_latest_execution(..., expected_first_name=...)` mit
Profil-Verifikation. Funktionsweise:

1. Polling auf Executions mit `id > baseline_id`.
2. Für jede gefundene success-Execution: Detail-API-Call, Extraktion von
   `first_name` aus Node 15 oder Node 03 Output.
3. Wenn `first_name` mit dem erwarteten matched → akzeptieren.
4. Sonst → in `rejected_ids` aufnehmen und weiter pollen.

**Effekt:**

- **Latenz-Resistenz**: 90s Pollzeit decken auch die Cold-Start-Lage mit drei
  Sheet-Loadern ab.
- **Profil-Sicherheit**: kein Mix-Up mehr möglich, selbst wenn mehrere Profile
  parallel triggern.
- **Status-Strenge**: nur `success` zählt als Treffer — der alte Bug aus Teil 2
  (Error-Executions als gültig akzeptiert) ist strukturell weg.

**Verifikations-Run nach Härtung:** Exec 200–206, **7/7 grün** mit eindeutiger
Zuordnung jedes Profils zu seiner Execution. Polling 38–47 pro Profil.

## n8n REST API — Erweiterung der Erkenntnisse aus Teil 2

Zusätzlich zum bekannten Top-Level- und settings-Whitelisting:

- **Neuen Node hinzufügen** funktioniert per PUT, wenn der Node ein gültiges Schema hat:
  `id` (UUID), `name` (eindeutig), `type`, `typeVersion`, `position` `[x,y]`,
  `parameters`, `credentials`. Genug Felder, dass `n8n` die Connectoren rendern kann.
- **Connections umverkabeln** geht direkt im `connections`-Object. Pattern für
  Sheet-Loader: `prevCodeNode → newLoader → consumerCodeNode`. Position des Loaders
  kann visuell oben/unten neben der Hauptlinie sein (z. B. `y=-64` während Hauptfluss
  auf `y=144` liegt).
- **typeVersion 4.7** mit `mode: "name"` ist die saubere Variante für Sheet-Referenzen
  per Tab-Namen (statt gid-Lookup). `cachedResultUrl` etc. sind optional.
- **Credentials** können wiederverwendet werden (gleiche `id` wie bestehende Sheet-Nodes).

## Wichtige Erkenntnisse

1. **Audit erst, migrieren später.** Phase 2a (Lese-Scan) hat gezeigt, dass die
   Node-12-Migration viel kleiner ist als befürchtet (0 statt N Sheet-Änderungen).
   Hätten wir blind angefangen, wäre Phase wesentlich aufwendiger gewesen.

2. **Toter Code findet sich beim Audit.** `ir_clinical_serum` war ein Special-Case
   für eine nie genutzte Filter-Variante. Migrations-Anlässe sind eine gute
   Gelegenheit, solchen Code mit zu entfernen.

3. **Generische Operatoren > Spezialfälle.** Der `<feld>=TRUE|FALSE`-Regex-Parser
   ersetzt nicht nur die zwei aktuellen Cases, sondern eröffnet einen ganzen
   Designraum für künftige Filter, ohne Code-Änderungen.

4. **Long-Format-Sheets skalieren am besten.** `regel_id | kategorie | … | aktiv`
   ist das Pattern, das sich für `map_conflict_rules`, `map_slot_rules`,
   `map_pflegelevel_scoring` und jetzt `map_priorities` durchzieht. Beibehalten.

5. **Baseline-Tabelle ist Pflicht.** Vor jeder Migration den Soll-Stand der 7 Profile
   einfrieren (Exec-IDs notieren), danach Vergleich. Bei jeder Abweichung sofort
   rollback per Backup-PUT.

6. **CSV-Import schlägt TSV-Paste.** Bei der `map_priorities`-Datenübergabe war die
   `.csv`-Datei (über "Datei → Importieren") deutlich angenehmer für den User als
   ein TSV-Block, der manuell in Spalten aufgesplittet werden muss.

## Was NICHT funktioniert hat

| Versuch | Warum |
|---|---|
| TSV-Block als Copy/Paste für Sheet-Befüllung | Tabs gehen in Markdown-Code-Blöcken oft verloren, alle Daten landen in einer Spalte. CSV-Datei + Import ist die richtige Variante. |
| Schreibzugriff per Service-Account auf Google Sheet | Bewusst **nicht** eingerichtet. Datenhoheit beim User halten, Sheet-Schreibvorgänge sind nicht atomar reversibel. |
| Erste Annahme "Node 12 braucht Sheet-Migration" | Phase 2a hat gezeigt: keine Sheet-Änderung nötig. Lese-Audit vor Schreib-Migration spart Aufwand. |
| Visual Studio Code zwischendurch einrichten | Nur initial-Setup erreicht (Extension, Login, Ordner-Öffnen). Quick-Open fand nichts (offenbar falscher Ordner / Indexing-Problem). Auf Folgesession verschoben. |
