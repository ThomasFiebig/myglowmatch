### 9. Test-Suite las falschen Node aus

Erst dachten wir, der Filter (Node 08) wäre umgebaut worden, weil
`applied_conflict_rules` im Output von **Node 15** leer war.

**Tatsächlich**: CON-12 feuerte sehr wohl. Sichtbar in Node 14's Output:

```json
applied_conflict_rules: [
  {"konflikt_id": "CON-09", "removed_products": ["fohncreme"], ...},
  {"konflikt_id": "CON-11", "removed_products": ["smoothing_*"], ...},
  {"konflikt_id": "CON-12", "removed_products": ["essig_shampoo", "essig_spuelung"], ...}
]
```

Aber Node 15 propagierte das Feld nicht weiter → leer in seinem Output.

#### Architektur-Erkenntnis (wichtig!)

- **Knoten-Nummerierung im Namen ist irreführend**: Im n8n-Canvas läuft der
  Datenfluss `… → 14 Konflikte auflösen → 12 Scoring & Slot-Befüllung →
  15 Routine sortieren`. Visuell liegt **Node 12 zwischen Node 14 und Node 15**.
- Wenn `applied_conflict_rules` von Node 14 nach Node 15 durchgereicht werden
  soll, muss **Node 12** das Feld im Output propagieren.

#### Patch (Thomas in Node 12, 2 Zeilen)

In Node 12's Return-Statement muss `applied_conflict_rules` aus dem Input
übernommen werden:

```js
// Pseudo-Code (genauer Patch in n8n-Canvas):
return {
  ...,
  applied_conflict_rules: $input.first().json.applied_conflict_rules || [],
  // (zweite Zeile war analog für ein weiteres Pass-Through-Feld)
}
```

Damit erscheint `applied_conflict_rules` auch im Output von Node 15 und in
nachgelagerten Nodes (17 Claude-E-Mail, 19 Log).

### 10. Test-Suite-Workaround (parallel zu Node-12-Patch)

Damit die Test-Suite unabhängig vom Workflow-Stand funktioniert, haben wir
`extract_routine_output()` erweitert:

- Basis-Output kommt aus `TARGET_NODE` (Node 15)
- Falls `applied_conflict_rules` dort leer ist, wird das Feld aus `CONFLICT_NODE`
  (Node 14) gemerged
- Marker `_conflict_source` wird gesetzt, damit der Konsument weiß, woher das
  Feld kommt

Neue Helper-Funktion `extract_node_json(detail, node_name)` extrahiert generisch
den JSON-Output eines beliebigen Nodes.

## Wichtige Erkenntnisse für die Zukunft

1. **Node-Nummerierung in den Namen ≠ Ausführungsreihenfolge.** Im Canvas läuft
   der Flow `… 14 → 12 → 15 …`, weil "12 Scoring" nach den Konfliktregeln
   einsortiert wurde. Wer Pass-Through-Felder mergen will, muss visuell schauen,
   nicht nur die Nummer im Namen.

2. **n8n-Webhook-Response-Mode `onReceived` blockiert Datenrückgabe** an den
   Caller — gut für schnelle Frontend-Antwort, aber Test-Tools müssen den Umweg
   über die Execution-API gehen.

3. **n8n-Cloud-API ist eventually consistent.** Direkt nach einem Trigger ist die
   neue Execution oft noch nicht in `GET /executions`. Polling mit
   `since_id > baseline` ist robuster als `since_timestamp`.

4. **`applied_conflict_rules` aus Node 14 holen ist robust**, weil das Feld dort
   garantiert erzeugt wird. Pass-Through-Bugs in nachgelagerten Nodes können das
   Feld verlieren — die Test-Suite umgeht das durch Mehrfach-Lookup.

5. **Score-Gleichstand zwischen Produkten → unbestimmtes Verhalten.** Statt
   einen Tie-Breaker im Scoring zu bauen, war hier der bessere Fix eine
   inhaltliche Konfliktregel (CON-12), die das "falsche" Produkt aus der
   Auswahl entfernt. Inhalt schlägt Sortier-Hacks.

6. **Mail-Nodes nicht ohne Test reaktivieren.** Wenn Mail-Nodes ihre Sender-Auth
   oder Body-Templates verloren haben, killen sie nicht nur ihre eigene
   Ausführung, sondern lassen den Status der Execution auf `error` springen
   — was nachgelagerte Logging-Nodes (`19`) ebenfalls nicht mehr erreicht.
